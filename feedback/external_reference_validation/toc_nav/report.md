# Tier 2 來源 A/B：TOC + 頁尾頁碼 → 獨立頁碼導航（報告草稿）

> 程式：[toc_extract.py](toc_extract.py)（VLM 讀 TOC）、[coverage.py](coverage.py)（頁尾對照 + 導航 + 覆蓋率）
> 與 [../vlm_first_last_results.md](../vlm_first_last_results.md)（來源 C）、[../../feedback/report_tier2.md](../../feedback/report_tier2.md) 平行

## 定位：為什麼需要「獨立頁碼」

來源 C（VLM 首尾段）要先知道「item N 在哪一頁」，才能把那頁餵給 VLM。原本這個頁碼是
**用 parser 抽出的內容去比對 PDF 得到的** → 導航與「被驗的內容」耦合：若 parser 把某 item 截斷或放錯，
導航會**跟著錯誤一起移動**，VLM 仍看到「parser 自己宣稱的那頁」而對得上 → 偽 PASS（§7.3 的截斷偵測因此偏樂觀）。

要打破這個耦合，需要**不依賴 parser 內容的頁碼來源**。10-K 本身就有兩個：
- **TOC**：文件自宣告的 `item → 印刷頁碼`（也是來源 B 的 item 清單/順序）。
- **頁尾印刷頁碼**：渲染頁底部的頁碼（也是來源 A 的頁碼單調訊號）。

兩者組合 → `item → 渲染頁索引`，全程不碰 parser 內容。

---

## 方法

1. **VLM 讀 TOC**（[toc_extract.py](toc_extract.py)）：掃前幾頁（TOC 可能跨頁），每頁丟 VLM；
   是 TOC 就逐列回 `item | 頁碼`，不是回 `NONE`。彙整得 `item → 印刷頁碼`。自我檢查：頁碼應**單調不遞減**。
2. **頁尾頁碼 → 渲染索引**（[coverage.py](coverage.py) `footer_map`）：每頁收集候選數字（字串尾端 + 底部 22% 的純數字 word），
   再用**單調序列擬合**鎖定真正的頁碼序列（先找「v, v+1」連續對定 offset，前後雙向沿著緩慢漂移的 offset 取最近候選；
   表格值/年份因不連續被排除）。缺頁的印刷頁用相鄰錨點**內插/外插**。
3. **合成 + 標題微調**：`item 印刷頁 →(對照)→ 渲染索引`，再在該頁附近用「Item N 標題」近頁首出現處微調 ±1。

### 設計（已定、本階段未整合測試）

- **信心 gate**：放行進來源 C 之前，全域 sanity（TOC 抓到且單調、頁尾序列存在、TOC 頁碼範圍 ⊆ 頁尾範圍）
  + **逐 item 標題對帳**（導航出的頁 ±1 必須真的有「Item N」標題，否則該 item 棄驗/退回內容比對）。
  → 只驗「對帳通過」的 item，§7.3 的截斷偵測就不再是「假設正確頁碼」的樂觀上界。
- **餵 VLM 的取頁**：head 給標題錨定的那 1 頁；tail 給 2 頁（邊界跨頁本質：N 的尾巴可能在 N+1 標題頁或其前一頁），
  prompt 一律以「Item N / Item N+1 標題」錨定，使多給頁也不會抄錯頁。

## 結果：5 份覆蓋率（model = gemini-3-flash-preview）

導航得到的 `item → 渲染頁`，對照資料集已知 start_page（內容比對推得）：

| filing | TOC 抓到 item | 完全命中 | ≤±1 |
|---|---|---|---|
| GDC_2023 | 23 | 15/16 | 16/16 |
| NFLX_2025 | 23 | 9/11 | 11/11 |
| RELL_2025 | 23 | 9/11 | 10/11 |
| TSLA_2023 | 23 | 11/12 | 11/12 |
| WMT_2026 | 23 | 12/12 | 12/12 |
| **合計** | **5/5** | **56/62 (90%)** | **60/62 (97%)** |

- **TOC 抽取 5/5 成功**（都 23 項，含 Part I–IV 全 item）。
- **導航 90% 完全命中、97% 在 ±1 內**，且**全程不依賴 parser 內容**。
- 解讀：這是兩個**獨立方法**（TOC+頁尾 vs 內容比對）的**互相佐證**——它們在 90% item 上指到同一頁，
  本身就是一種三角驗證；殘差 ±1~2 由取頁 window + 標題錨定吸收。

### 過程中解掉的三個問題

| 問題 | 成因 | 修法 |
|---|---|---|
| NFLX/TSLA「抓不到 TOC」 | VLM 其實讀對了（`Item 1. | 1`），是解析 regex 不允許數字後的句點 | regex 加 `\.?` |
| GDC 頁尾 vs RELL 頁尾顧此失彼 | 頁碼版面不同：GDC 在字串尾端、RELL 被排在字串中段（y=83% 卻非尾端） | 改**單調序列擬合**（位置 + 連續性），不靠單頁猜 |
| RELL 導航全錯 | RELL offset≈0，但 item 起始的印刷頁（4/9/18）那幾頁剛好沒抽到頁尾數字 | 錨點**內插/外插**補出缺頁 |

## 誠實邊界

1. **filing-dependent 前提**：需要「TOC 有頁碼欄」+「渲染頁有可抽取的印刷頁碼」。5 份在強化抽取後皆可；
   但純 flow、無頁碼的 filing 此招不適用 → 需退回內容比對。
2. **VLM 讀 TOC 會錯**：用「頁碼單調」+「逐 item 標題對帳」兩個 deterministic 檢查約束（讀錯會被攔）。
3. **頁尾抽取靠序列擬合 + 內插**：處理了 offset 漂移與缺頁，但極端版面仍可能失敗。
4. **導航 ±1~2 殘差**：靠取頁 window + 標題錨定吸收；對「該給 VLM 看哪頁」已足夠。
5. **驗證基準是「內容比對推得的 start_page」**：故「命中」= 兩獨立方法一致，非對照人工視覺頁標註。
6. **gate / 取頁 window 為設計，尚未整合成端到端 boundary 重測**（本階段先完善閉環，不再加測）。
7. 樣本為 5 份。
